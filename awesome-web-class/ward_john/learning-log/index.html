  • CSS vendor prefixes allow browsers to add support for new CSS features. Examples: webkit, moz Box sizing When setting the width you need to account for padding and border size. border-box: width and height is measured with content, padding and border content-box: width and height is measured with content only More Info Display: none vs Visibility: hidden Display: none will render the page as if the element didn't exist. Commonly used with javascript. Visibility: hidden will hide the element but still take up the space Learn Layout Use max-width to handle small windows so there is no scroll bar is the width is bigger then the browser. Reading Response
  • Semantics in HTML 5
  • Following A Web Design Process
  • How to Read a Wireframe

John Allsopp, in "Semantics in HTML 5" talks about the past of HTML and the future, and predicts that HTML will be around for many years to come. HTML 5 made a lot of changes to the language that allowed it to cope with the present demands for rich content on the web. Then he goes on to theorize about several methods of how we can improve HTML in the future without breaking backwards compatibility. The future portion of it seemed a bit silly and theoretical to me; I don't think this guy can predict the future or what sort of content will be necessary for the web to support.

Luke Reimer's article, "Following A Web Design Process," is all about the actual process that a professional web developer might follow in creating a website, from planning to implementation to post-deployment. He has a lot of great points in his instructions, as well as resources to help out. I could see this being particularly useful to a freelance web developer starting out, so they have a disciplined process to follow; of course it could be the same for a traditionally-employed web developer, but where a firm might have their own structured process for an employee to learn, a freelancer would be better off following this than just making it up as they go.

I don't think "How to Read a Wireframe" warrants a response. Cool picture.

In The Unproven Path, Joel Spolsky talks about how he abandoned his methodical method of programming in creating Stack Overflow, and he is nonetheless thrilled with how it turned out. How can it be, he wonders incredulously in the article, that he could do everything "wrong" and yet still have the project turn out right? Reading Response
  • PROGRAMMING IS NOT A CRAFT
  • Why Software Development is a Craft
  • Software Engineering: Dead?

To me, craft means taking pride in one's work. Being inspired to create the most elegant solutions to problems, rather than to just hack things together. Engineering makes do with what already exists; progress is made through craft. Jeff Atwood articulated this the best of the three readings. Doug Bradbury took a much more metaphorical approach in comparing software development to woodworking and other crafts and put forth the idea that with a craft, something might have intrinsic value separate from its usefulness. With software development, I don't see that intrinsic value as necessary for craftsmanship, though I think it is a possible outcome of craftsmanship. Dan North blatantly rejected this notion that craftsmanship is important at all and in fact even proposed that it is detrimental in software development.
